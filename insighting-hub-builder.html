04 / The rhythm

    
This is where it compounds.

    
Building the jobs is the easy half. Running them on a rhythm is what makes the ledger worth reading. Jobs one and two run weekly or monthly. Job three runs every day, on its own.

    

      
RUN
Run it on schedule
Same board, same day, same cohort size. The agent earns trust through rhythm, like people do.

      
READ
Read every finding
All of it, like an intern's first report. Never skim, never auto-trust a confidence score.

      
CORRECT
Correct the brief
Never the finding. One edit to the instructions kills that mistake forever.

      
FEED
Feed every decision
Point your other prompts at the same ledger. That's the whole trick, no memory feature required.

    

    
and the corrections feed the next run. That's the compounding.

    

      The one rule people break
      
The ledger is not a report you read once a quarter and forget. It only works if it feeds the same document you use to write the next script, brief the next launch, or fix the next thing. A dashboard nobody else's workflow reads is a graveyard, not a system.

    

  






  

    
05 / Where these systems die

    
Four mistakes. All avoidable.

    

      

        
        
Proactive listening dies the moment customers feel farmed. Define a large enough cohort and rotate through it, and cap contact per person, once a month is plenty for most categories. A smaller, resentful list is worse than no list.

      

      

        
        
A new destination nobody remembers to check is worthless. It only works wired into a workflow you already run daily, your planning doc, your marketing brief, your product review. Point existing habits at it instead of creating a new one.

      

      

        
        
Inconsistent formats multiply agent errors. One board, one row shape, three job types. The agent should never have to guess which fields matter this week.

      

      

        
        
"Customers seem unhappy" is not a finding. "62% cite no lock-in as the reason they bought" is. Same rule as hiring any agent: if you can't say whether the output did the job, you can't manage it.
